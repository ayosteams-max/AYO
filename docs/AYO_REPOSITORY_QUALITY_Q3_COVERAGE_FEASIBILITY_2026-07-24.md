Date: 2026-07-24

## Executive conclusion

The authoritative whole-`BACKEND` combined line-and-branch result is **60.6977%**
(displayed as 61%), against an exact 70.00% gate. Reaching the gate requires
**2,440 additional covered measured elements** with the denominator unchanged.

Meaningful deterministic tests can still improve Identity, Settlement, Identity
runtime, Field Operations and several application boundaries. They are unlikely
to close the entire gap honestly. The likely non-PostgreSQL ceiling is **66.42%**;
the optimistic defensible ceiling is **68.89%**. Reaching 70% without PostgreSQL
would therefore probably require SQL repositories to be driven by broad mocks or
other execution-only tests that do not prove their actual contracts.

The recommended next bounded step is to **seek governance approval to begin the
disposable PostgreSQL 17/PostGIS 3.6 baseline while Q3 remains open**. This is not
a request to lower or reorder acceptance gates. It restores the evidence sequence
originally recommended by the Repository Quality Initiative.

## Surface classification

### A — meaningfully coverable now

- Remaining public Identity Account Access lifecycle and denial paths.
- Settlement reconciliation, approval, hold and exception application rules.
- Identity runtime registration, refresh and recovery validation using bounded
  repository protocols.
- Field Operations case/review authority and failure behavior.
- Payment and Refund application-level validation and non-disclosure.
- Pure deterministic serialization, hash, projection and transition helpers.

Estimated defensible opportunity: **1,000–2,150 combined elements**, with rapidly
declining value near the upper bound.

### B — PostgreSQL-backed testing required

- SQL repository reconstruction, uniqueness, locking and not-found semantics.
- Real optimistic concurrency and competing writers.
- Transaction rollback across aggregate, evidence, audit, outbox and idempotency.
- Immutable database triggers and UPDATE/DELETE rejection.
- Migration, index, constraint, grant and least-privilege behavior.
- Restart durability and backup/restore.

The current run skipped **201 PostgreSQL-dependent tests** because
`AYO_TEST_DATABASE_URL` was unavailable. Static inventory shows 4,014 uncovered
persistence elements. A defensible pre-execution estimate is that the existing
database suite could add roughly **1,600–2,600 combined elements**, including
application paths entered through integration tests. Only execution can replace
that estimate with evidence.

### C — external integration or unavailable environment

- Provider-authenticated payment behavior beyond the provider-neutral envelope.
- External maps/PostGIS spatial execution, operational messaging and other
  deliberately unimplemented adapters.
- Production-only activation and infrastructure behavior.

These surfaces must not be mocked into certification.

### D — governance or contract decision

- Authorization to execute the approved disposable PostgreSQL baseline while Q3
  remains open.
- Approval of a clean reviewed commit/worktree to which certification evidence can
  be bound.
- Any proposal to remove genuinely dead or superseded code from the denominator.

No threshold or scope change is recommended.

### E — low-value/artificial coverage

- Broad mocked SQLAlchemy result chains whose assertions merely repeat configured
  mock values.
- Import-only protocol/contract tests.
- Calling route wrappers solely to execute dependency wiring already covered at
  the application boundary.
- Tests for prohibited production activation or unimplemented external providers
  that invent behavior.

maintaining the mission’s test-quality rules**. It is technically possible to
execute more lines with mocks, but that would confuse coverage with evidence.

## Sequencing authority and contradiction

AYO-RQC-1 requires both 70.00% coverage and PostgreSQL certification, but does not
state that coverage must pass first:

- `AYO_REPOSITORY_QUALITY_CONTRACT_PROPOSAL_2026-07-24.md`, lines 45–47,
  defines coverage and PostgreSQL as separate mandatory controls.
- The same document, lines 163–166, provisions PostgreSQL before treating
  database skips and sub-70 coverage as failures in the unified CI job.
- `AYO_REPOSITORY_QUALITY_INITIATIVE_2026-07-24.md`, lines 222–233, places the
  PostgreSQL baseline before broad coverage work and explicitly says this is
  deliberate because the 201 skipped tests may cover the apparent gap.
- The Q3 mission checkpoints in `AYO_ENGINEERING_WORKFLOW.md`, lines 748–782,
  prohibited PostgreSQL only within those bounded missions.

Therefore, “Q3 must pass 70% before PostgreSQL certification may begin” is **not an
approved contract rule or technical necessity**. It is a later mission-ordering
assumption, and it conflicts with the evidence-based sequence in the approved
quality initiative. Current PostgreSQL execution is still unauthorized because
each Q3 mission explicitly stopped before it; this assessment does not expand that
authority.
